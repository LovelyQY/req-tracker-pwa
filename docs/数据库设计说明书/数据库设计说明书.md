# 数据库设计说明书（req-tracker-pwa 需求跟踪 PWA）

> 版本：v1.3.48（基线）
> 本文描述系统持久化设计：两个 IndexedDB 库（主业务库 + 媒体库）的全部数据表、字段、索引、关系与约束，以及**待开发权限系统的 4 张新表**与历史表（SCD-2）设计。
> 字段级权威定义以 `DB_SCHEMA.md` 为准；权限表结构以 `plans/权限功能_PLAN.md` §2 为准。

---

## 1. 引言

### 1.1 目的
明确 req-tracker-pwa 的数据库架构、各表结构、主外键关系、索引策略与数据完整性机制，作为数据层实现与评审依据。

### 1.2 数据库总览
系统使用 **两个相互独立的 IndexedDB 数据库**：

| 数据库 | 版本 | 用途 | 表数量 |
|---|---|---|---|
| `req-tracker` | base 3（运行时可自增） | 主业务数据（公司/部门/职位/人员/项目/版本/字典/需求任务/待办/流程/更新日志 + 权限 4 表） | 12 + 4（待开发） |
| `req-tracker-pwa` | 4 | 图片 / 附件二进制（Base64 dataURL），规避主库膨胀 | 2 |

### 1.3 全局约定
- **主键**：所有表统一 `keyPath: 'id'`，值为 `RT_DB.genId()` 生成的 **32 位十六进制串**（定长、无碰撞语义）。
- **审计字段**：`createdBy / createdAt / updatedBy / updatedAt`（写入人 / 时间戳），种子数据 `createdBy='system'`。
- **字典 code**：状态/类型/优先级等仅存 code（如 `REQ`/`HIGH`/`TODO`），展示文案取自 `dict` 表，实体不冗余存储中文。
- **图片/附件只存短 ID**：主表仅保留 `imageIds`/`attachmentIds` 数组，真实字节存于媒体库，保持记录"轻"。

## 2. 主业务库 `req-tracker`

统一由 `db.js` 拥有（`RT_DB.openDB()`），各数据模块通过 `RT_DB.registerStore()` 注册 store 与索引。

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | string(32hex) | 主键 |
| `companyName` | string(1–50) | 公司名称 |
| `companyType` | string | `总公司` / `分公司` |

### 4.3 `role_permission`（角色-权限关系表，历史，append-only）
| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | string(32hex) | 主键 |
| `roleId` | string(32hex) | → roles.id |
| `menuCode` | string | **关系主键/判定键** → menus.menuCode |
| `menuId` | string(32hex) | → menus.id（镜像，引用完整性） |
| `snapshotId` | string(32hex) | 一次保存的分组标识（同批共享） |
| `createdBy/At` `updatedBy/At` | string/number | 审计（追加写时=created*） |

- 索引：`roleId`、`menuCode`、`snapshotId`、`updatedAt`
- **写入规则**：仅追加，保存时生成新 `snapshotId` 批量插入，旧行不 UPDATE/DELETE；`roles.menuCodes` 同步覆盖为当前集。

### 4.4 `user_role`（人员-角色关系表，历史，append-only）
| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | string(32hex) | 主键 |
| `userId` | string(32hex) | → users.id |
| `roleId` | string(32hex) | → roles.id |
| `snapshotId` | string(32hex) | 一次分配的分组标识 |
| `createdBy/At` `updatedBy/At` | string/number | 审计 |

- 索引：`userId`、`roleId`、`snapshotId`、`updatedAt`
- **写入规则**：同 4.3 追加写；`users.roleIds` 同步覆盖为当前集。

### 4.5 历史表 SCD-2 双轨设计
- **历史表（role_permission / user_role）只作审计日志**：保存时**仅追加新行**（带新的 `snapshotId`），永不更新/删除旧行 → 满足"保留历史、只新增"。
- **当前态去范式化**：`roles.menuCodes` / `users.roleIds` 在每次保存时整体覆盖为最新集 → 运行时快速解析，不回查历史表 JOIN。
- **查询历史**：按 `roleId + snapshotId` / `userId + snapshotId` 回溯任意历史快照（审计用）。

## 6. 索引策略
- 按查询路径建索引；所有外键字段建索引。
- 多值字段（`developerIds`/`relatedDevIds`）用 `multiEntry: true` 支持按元素查询。
- 历史表按 `roleId`/`userId`/`snapshotId`/`todoId`/`taskId` 建索引，支撑按主体与时间回溯。
- 列表默认按 `updatedAt` 排序。

---

## 7. 数据完整性设计
- **主键唯一性**：32 位 `genId()` 随机生成，碰撞概率可忽略；业务唯一性（account/employeeNo/roleName/menuCode）在 `validateX` 中校验。
- **字典 code 合法性**：写入时 `assertDictCode` 校验 code 属于对应字典枚举。
- **外键存在性**：先跨连接异步校验（项目/版本/人员/公司/部门），再开写事务，避免死锁。
- **审计**：所有写操作记录 `createdBy/updatedBy`（取 `getCurrentUserAccount()`）。
- **级联删除**：删需求任务/待办时级联清理图片、附件与流程流水，避免孤儿数据。
- **历史可追溯（权限）**：关系表 append-only + `snapshotId`，当前态去范式化，兼顾审计与性能。
- **软隔离边界**：数据权限为前端只读过滤（无后端），非服务端鉴权（计划中已明示）。

## 8. 字典枚举（种子数据，只读参考）
