## 1. 引言

### 1.1 目的
明确 req-tracker-pwa（一款**纯前端、离线优先的需求跟踪 PWA**）的功能范围、用户场景、业务规则与约束，作为设计、开发、测试与验收的依据。

### 1.2 范围
- **已上线功能**：登录与会话、首页需求看板（需求任务 + 待办）、需求任务管理、待办管理、基础数据管理（公司/部门/职位/人员/项目/项目版本/字典）、统计报表、个人中心（个人信息/账号安全/我的状态）、存储与备份、关于与更新日志。
- **待开发功能**：权限系统——角色管理、权限管理、人员-角色分配、数据权限（部门隔离）、系统管理员默认角色，以及"新增页面/按钮必须登记权限"的强规则。

### 1.3 术语与定义
| 术语 | 说明 |
|---|---|
| PWA | 渐进式 Web 应用，可安装到主屏、离线可用 |
| 需求任务 | 首页看板"任务"区的主体，归属项目/项目版本，有完整生命周期 |
| 待办 | 首页看板"代办"区的主体，按类型分任务事项/缺陷追踪/会议 |
| 字典 | 系统枚举（任务类型/优先级/状态等），仅代码维护、页面只读 |
| 32 位 ID | `RT_DB.genId()` 生成的 32 位十六进制串，作所有表主键 |
| 强规则 | 权限功能上线后，新增页面/按钮必须登记权限的强制约定（见 §4.7） |

### 1.4 参考资料
- `DB_SCHEMA.md`（数据结构）、`RULES.md`（发版与编码规则）、`plans/权限功能_PLAN.md`（权限设计）、`CHANGELOG.md`（版本历史）。

## 2. 产品总体描述

### 2.1 产品定位
面向产品/测试/开发团队的个人/小团队协作工具：在移动端和桌面浏览器中**离线记录需求任务与待办、跟踪状态流转、按项目/类型/状态统计**，无需后端服务器。

### 2.2 用户特征与角色
- **当前**：所有登录用户权限对等（仅 `admin` 拥有全部数据，因无后端、数据各设备独立）；无细分角色。
- **待开发（§4）**：引入角色（如系统管理员、项目经理、测试等），按角色授予页面与按钮权限；`admin` 默认拥有"系统管理员"角色。

### 2.3 运行环境
- 现代浏览器（Chrome/Edge/Safari/华为浏览器等），支持 IndexedDB、Service Worker、Cache API。
- 通过 HTTPS 或 localhost 访问（密码加密 `crypto.subtle` 需安全上下文）。
- 可"添加到主屏"作为 PWA 安装运行。

### 2.4 设计与实现约束
- **单页 + 多 HTML 入口**：各功能以独立 `.html` 实现，共享 `auth.js`/`db.js`/`config.js` 等全局模块。
- **无后端、本地优先**：全部业务数据存于浏览器 IndexedDB；多设备数据各自独立，无云端同步。
- **主键与 ID**：所有表主键为 32 位十六进制 `genId()`；外键存 32 位 ID；ID 类长度用专用 `*_ID_MAX`（禁复用人工字段上限）。
- **发版规则**：推 `main` 必须 bump 版本（`release.sh`），`pre-push` hook 强制；静态资源引用须带 `?v=` 缓存破坏；计划文档入 `plans/` 并提交。

### 2.5 假设与依赖
- 用户自行负责数据备份（存储与备份页导出）。
- 字典枚举由代码播种，页面只读。

## 3. 现有功能需求（已上线）

### 3.1 登录与会话管理
- **登录**：支持「账号」或「工号」+ 密码登录（`login/classic.html`）；密码以 SHA-256 存储与校验。
- **记住登录**：勾选 → 会话存 `localStorage`（关浏览器仍保留，受过期时间约束）；不勾选 → 存 `sessionStorage`。
- **登录时长**：提供 24 小时（默认）/ 7 天 / 30 天，写入 `exp = now + 天数×24h`。
- **会话校验**：`getSessionAccount()` 校验 `exp`，过期清除并跳登录页；手动退出立即清除。
- **默认管理员**：首次运行 `ensureDefaultAdmin()` 幂等播种 `admin / 123`，保证可直接登录。
- **返回栈**：登录成功清空返回栈，避免带旧链路返回。

### 3.2 首页需求看板（index.html / app.js）
看板明确区分**「任务」区**与**「代办」区**两块。

#### 3.2.1 需求任务区
- 卡片展示：任务名称、状态、优先级、类型、所属项目/版本、开发人员、状态流转时间线（灰显单行）。
- 筛选：按类型 / 状态 / 优先级 / 项目 / 关键字过滤。
- 操作按钮按状态动态显示（见 §3.3 生命周期）。
- 顶部"新建需求任务"入口。

#### 3.2.2 待办区（三类型）
- 类型切换：**任务事项**（TASK_ITEM）/ **缺陷追踪**（BUG）/ **会议**（MEETING），各类型卡片字段布局不同：
  - 任务事项：开发人、开始/完成时间。
  - 缺陷追踪：关联任务、反馈人/时间。
  - 会议：会议时间、地点。
- 操作按钮按类型与状态动态显示（见 §3.4）。
- 顶部"新建待办"入口，按当前类型打开对应表单。

### 3.3 需求任务管理
**数据层**：`requirement-tasks.js`，store `requirementTasks`。

- **字段**：
  - `taskName` 任务名称（1–100，必填）
  - `taskDesc` 任务描述（0–1000）
  - `taskTypeCode` 任务类型（字典 `任务类型`，必填）
  - `priorityCode` 优先级（字典 `优先级`，必填）
  - `statusCode` 任务状态（字典 `任务状态`，必填）
  - `projectId` 所属项目（→ `projects`，必填）
  - `projectVersionId` 所属项目版本/需求组（→ `projectVersions`，选填，须归属所选项目）
  - `developerIds` 开发人员（→ `users`，支持多选）
  - `zentaoId` / `zentaoSubId` 禅道 ID / 子 ID（外部系统关联，选填）
  - `imageIds` / `attachmentIds` 图片 / 附件短 ID（真实字节存于媒体库 `req-tracker-pwa`）
  - 生命周期：`devSubmitTime/By`、`testStartTime/By`、`testEndTime/By`、`onlineTime/By`
  - 审计：`createdBy/At`、`updatedBy/At`
- **CRUD**：创建/编辑/删除；删除时**级联清理**关联图片、附件与任务生命流程记录（`task-lifecycles`）。
- **校验**：字段格式 + 字典 code 合法性 + 外键存在性（项目/版本/开发人员）。
- **生命周期（TASK_OPERATION，8 个）**：`DEV_SUBMIT` 开发提交 / `TEST_START` 测试开始 / `PAUSE` 暂停 / `RESUME` 暂停恢复 / `TEST_DONE` 测试完成 / `ONLINE` 上线 / `RESET` 重置 / `DELETE` 删除。流转记录存 `task-lifecycles`（append-only）。

### 3.4 待办管理
**数据层**：`todos.js` + `todo-lifecycles.js`，store `todos` / `todoLifecycles`。

- **三类型**（字典 `TODO_TYPE`）：任务事项（TD_*）、缺陷追踪（BUG_*）、会议（MT_*）。
- **字段按类型差异**：任务事项含开始/完成时间、开发人；缺陷追踪含关联任务、反馈信息；会议含会议时间、地点、取消原因。
- **生命周期（TODO_OPERATION）按类型与状态动态显示**（`app.js` `getTodoActions`，行 2632）：
  - 任务事项（TD_TODO/TD_DOING/TD_DONE）：开始处理、完成、编辑、删除、重置
  - 缺陷追踪（BUG_TODO/DOING/DONE/WAIT_DEV/ONLINE）：开始处理、完成、**转交**、**上线**、编辑、删除、重置
  - 会议（MT_NOT_STARTED/IN_PROGRESS/ENDED/CANCELLED）：开始、取消、结束、编辑、删除、重置
  - 说明：三者按钮**不同**——任务事项无转交/上线/取消/结束；缺陷无取消/结束；会议无完成/转交/上线。
- **流转流水**：每次状态操作写 `todoLifecycles`（`todoId`/`statusCode`/`operationCode`/`operator`/`operateTime`，append-only），用于渲染时间线与状态标签。

### 3.5 基础数据管理（basic-data.html）
入口页罗列以下模块，下钻进入各管理页：

| 模块 | 关键业务规则 |
|---|---|
| 公司管理 | 总公司 / 分公司；分公司 `parentId` 指向总公司；删除前须删下级 |
| 部门管理 | 归属公司；自引用层级（支持多级），防环；删除前须删下级 |
| 职位管理 | 职位名称 / 编码 / 职级（职级 code 取自字典） |
| 人员管理 | 工号 / 姓名 / 部门 / 职位 / 人员状态；`account` 自动取工号，默认密码 `sha256("123")` |
| 项目管理 | 归属部门；名称 / 描述 / 状态 |
| 项目版本管理 | 归属项目（需求组）；名称 / 描述 / 状态 |
| 字典管理 | 系统枚举**只读**（任务类型/优先级/任务状态/任务操作管理/项目状态/人员状态/职级/代办类型/代办事项状态/缺陷追踪状态/会议状态/代办操作），仅代码播种 |

> 待开发：基础数据页将新增**角色管理**、**权限管理**入口（见 §4）。

### 3.6 统计报表
- **汇总页** `report.html` + **四个子页**：任务统计（`report-task`）、缺陷统计（`report-bug`）、待办统计（`report-todo`）、会议统计（`report-meeting`）。
- 基于筛选条件（项目/类型/状态/时间等）统计数量与分布；共享渲染逻辑 `report-common.js`。

### 3.7 个人中心
- **个人信息**（`profile.html` / `profile-edit.html`）：昵称、账号、密码、手机、邮箱、标签、个性签名、头像。
- **账号与安全**（`security.html`）：账号安全相关设置（密码等）。
- **我的状态**（`status.html`）：状态选项 无状态 / 忙碌中 / 会议中 / 出差中，抽屉头像区展示。

### 3.8 存储与备份（storage-backup.html）
- 导出全部 IndexedDB 业务数据为备份文件；导入恢复，便于换设备/防丢。

### 3.9 关于与更新日志（about.html）
- 展示关于信息、版本号、`CHANGELOG.md` 更新日志、文档入口。

## 4. 待开发功能需求：权限系统（RBAC + 数据权限）

> 详细设计、表结构、分批清单见 `plans/权限功能_PLAN.md`。本节仅列功能需求。

### 4.1 总体目标
在现有 PWA 上增加一套**角色（Role）/ 权限（Permission）/ 人员-角色分配 / 数据权限（部门隔离）**体系，实现"谁能看什么页面、能点什么按钮、能看到哪些数据"的精细控制；`admin` 默认拥有系统管理员角色、具备全部权限。

### 4.2 角色管理（`role.html`）
- 角色列表：角色名称、启停状态、权限数量。
- 新增 / 编辑角色：填写**角色名称**，勾选权限。
- **仅可勾选「已配置」的权限**（未与代码注册表对应的权限禁用勾选）。
- 启停角色；系统管理员角色不可停用 / 不可删除。
- **删除约束**：仅当**无任何人员引用**该角色时可删除（D4）；删除前展示引用人数，有引用则禁用删除。
- **展示引用的人**：列表/详情展示引用该角色的人员（工号/姓名，隐藏 32 位 ID），点击可查看引用人员清单，便于审计。

### 4.3 权限管理（`permission.html`）
- 以**树形结构**展示「页面 + 操作按钮」权限节点（模块 > 页面 > 按钮）。
- 新增 / 编辑 / 启停权限节点。
- **配置状态标识**：每个节点显示
  - **已配置**（代码真正守卫）：权限 code 存在于代码注册表 `PERMISSION_REGISTRY`；
  - **新增未真正配置**：节点存在于数据但 code 未接入代码守卫，不生效（灰/警示徽标）。
- 权限节点可**启停**：停用后该权限全局不生效（高于角色分配，见 §4.6 停用优先）。

### 4.4 人员-角色分配（`user.html`）
- 人员表单新增「分配角色」多选（仅启用角色可选）。
- 保存时写入人员-角色关系（历史追加，见 §4.8），并覆盖人员当前角色集。
- 人员列表展示已分配角色名称（隐藏 32 位 ID）。

### 4.5 数据权限（部门隔离）
- 人员仅能看到**本部门及下级部门**的数据（部门子树，含下级）。
- 过滤在数据层读取时生效（项目/任务/人员/部门/报表等列表）。
- **管理员豁免**：`admin` / 系统管理员角色跳过部门过滤，可见全部数据（最高权限）。

### 4.6 系统管理员默认角色与「停用优先」
- 首次运行幂等播种「系统管理员」角色（拥有全部权限），并绑定 `admin` 账号。
- **停用优先**：权限节点停用（`enabled=false`）后，无论角色是否勾选，该权限全局不生效——是比角色分配更高优先级的全局开关。

### 4.7 强规则（长期生效）
自权限功能上线起：**任何新增的 HTML 页面或操作按钮，必须在 `PERMISSION_REGISTRY` 登记对应权限码、写入 `menus`、并用 `data-perm`/`RT_PERM.can` 接线**。发版自检将扫描 `data-perm` 取值，未登记注册表则阻断发版。本规则写入 `RULES.md`，与"发版必 bump 版本"同级强制。

### 4.8 数据模型概览（四张新表）
| 表 | 关键字段 | 说明 |
|---|---|---|
| `roles` | id, roleName, menuCodes[](去范式化当前权限), isSystemAdmin, enabled, 审计 | 角色 |
| `menus` | id, menuCode(业务编码/判定键), menuName, parentCode(→父 menuCode), nodeType(module/page/op), enabled, 审计 | 权限/菜单树（D1 用 menuCode 关联） |
| `role_permission` | id, roleId, menuCode(+menuId), snapshotId, 审计 | 角色-权限关系，**追加写历史** |
| `user_role` | id, userId, roleId, snapshotId, 审计 | 人员-角色关系，**追加写历史** |

- 历史表 append-only（保留历史、只新增）；当前态去范式化在 `roles.menuCodes` / `users.roleIds`，运行时快速解析。
- 「已配置」以 `PERMISSION_REGISTRY` 为唯一真相。

### 4.9 分批实施路线（详见计划）
- **阶段 0（数据层/核心）**：批次 81 权限数据层 → 82 注册表+菜单种子+已配置判定 → 83 运行时 `RT_PERM` → 84 系统管理员播种。
- **阶段 1（管理页）**：85 基础数据入口 → 86 角色管理 → 87 权限管理 → 88 人员-角色分配。
- **阶段 2（按钮级强制）**：89 守卫机制+看板接线 → 90 基础数据页接线 → 91 报表/个人/系统页接线+抽屉收起。
- **阶段 3（数据权限）**：92 数据权限核心 → 93 报表过滤+一致性核验。
- **阶段 4（收尾）**：94 文档+全量单测+发版（并将强规则写入 `RULES.md`）。

## 5. 非功能性需求

### 5.1 离线可用 / PWA
- `manifest.json` + `sw.js` 提供安装与离线缓存；新版本经 `fetch(..., {cache:'no-store'})` 拉取最新 HTML。
- 静态资源引用带 `?v=` 缓存破坏，避免命中旧缓存。

### 5.2 性能
- 本地 IndexedDB 读写；权限解析去范式化（`menuCodes`/`roleIds`）+ 会话缓存，避免每次渲染扫历史表。
- 列表按更新时间排序，大数据量下保持响应。

### 5.3 安全
- 密码 SHA-256，不存明文。
- 会话带过期时间。
- **权限/数据权限为前端软隔离**：因无后端，数据权限是前端只读过滤，不能替代服务端鉴权；对直接读 IndexedDB 的恶意用户无效（计划中已明示边界，不伪装成硬安全）。

### 5.4 兼容性
- 移动端优先；去除点击蓝色高亮（`-webkit-tap-highlight-color`）、保留键盘 `:focus-visible` 焦点。
- 返回用 `goBack()`（返回栈）而非 `history.go(-1)`，兼容华为等浏览器。

### 5.5 可维护性与发版
- 每次发版 bump 版本（`release.sh` 同步 7 处版本号 + 生成 `CHANGELOG.md`）。
- `pre-push` hook 强制版本升级与日志非空。
- 计划/执行清单入 `plans/` 提交，可追溯。

## 6. 数据模型概览（引用 DB_SCHEMA.md）
现有 12 张表：`companies` / `positions` / `departments` / `users` / `projects` / `projectVersions` / `dict` / `requirementTasks` / `taskLifecycles` / `todoLifecycles` / `changelog` / `todos`（媒体库 `images`/`attachments` 在独立库 `req-tracker-pwa`）。
权限功能新增 4 张表：`roles` / `menus` / `role_permission` / `user_role`（见 §4.8）。
完整字段、索引与约束以 `DB_SCHEMA.md` 为准。
