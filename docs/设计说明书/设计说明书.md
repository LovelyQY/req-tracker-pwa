## 2. 总体设计

### 2.1 技术架构
- **无后端、纯前端 PWA**：所有业务逻辑与持久化均在浏览器端完成，无服务器 API。
- **分层**：
  - 持久层：`db.js` 统一拥有主业务库（`req-tracker`），`imgstore.js` 拥有媒体库（`req-tracker-pwa`）；各数据模块通过 `RT_DB.registerStore()` 注册自己的 store 与索引。
  - 数据模块层：每个实体一个 `*.js`（如 `companies.js`/`departments.js`/`users.js`），暴露 `RT_XXX` 全局 API（CRUD + 校验 + 查询）。
  - 页面层：各功能独立 `.html`，引入共享全局脚本后渲染。
  - 离线层：`sw.js` + `manifest.json` 提供 PWA 安装与离线缓存。
- **单一事实来源**：`config.js`（`RT_CONFIG`）收口所有 IndexedDB 库名/版本/store 列表，避免硬编码。

### 2.2 运行环境与部署
- 部署于 GitHub Pages（静态托管），经 HTTPS 访问；`crypto.subtle`（密码加密）要求安全上下文。
- 发版即推送 `main` + `release.sh` 升级版本号 + 重建 Pages；CDN 边缘有 1–10 分钟滞后，以 `git`/`raw` 文件为真相。

### 2.3 模块 / 目录划分
```
根目录：各功能 .html（index / basic-data / company / department / position /
        user / project / project-version / dictionary / profile* / security /
        status / report* / storage-backup / about / login）
共享脚本：db.js config.js auth.js sw.js manifest.json
数据模块：companies.js departments.js positions.js users.js projects.js
        project-versions.js dictionary.js requirement-tasks.js task-lifecycles.js
        todos.js todo-lifecycles.js changelog.js （+ 待开发 permissions.js / permissions-registry.js）
测试：tests/test-batch*.js
计划：plans/*.md
文档：docs/需求说明书 设计说明书 数据库设计说明书 测试用例 用户使用手册
```

### 2.4 全局设计约定
- **主键与 ID**：所有表主键 `id` 为 `RT_DB.genId()` 生成的 **32 位十六进制串**；外键存 32 位 ID；ID 类长度用专用 `*_ID_MAX: 64`（禁复用 `EMPLOYEE_NO_MAX`/`ACCOUNT_MAX`）。
- **审计字段**：`createdBy/createdAt/updatedBy/updatedAt`，写入用 `getCurrentUserAccount()`，种子用 `'system'`。
- **发版与缓存**：静态资源引用带 `?v=<版本>` 缓存破坏；`release.sh` 末尾全站 `?v=` 漂移自检；`pre-push` hook 强制版本升级与日志非空。
- **导航约定**：下钻用 `navTo(url)`（压返回栈），返回用 `goBack()`（弹栈回上一页），不依赖 `history.go(-1)`/referrer（兼容华为等浏览器）。
- **登录闸门**：各页 `<script>` 段调用 `getSessionAccount()`，无效则跳登录页。

## 3. 基础设施模块设计

### 3.1 数据层 `db.js`
- 统一拥有主库 `req-tracker`，版本 `DB_VERSION_BASE=3`，运行时取 `max(base, 探测到的已有版本)` 并随缺失 store/索引自增，避免版本冲突。
- 各模块加载时 `registerStore(name, {keyPath, indexes, onUpgrade})` 注册；首次打开即创建全部已注册 store；跨页面懒注册时自动补齐。
- `genId()`：16 字节随机数 → 32 位十六进制小写串（固定长度）。

### 3.2 配置中心 `config.js`
- `RT_CONFIG.databases` 定义 `main`/`media` 库的 `name/version/stores/owner`，供 `db.js`/`imgstore.js` 同步读取，消除硬编码。

### 3.3 会话与导航 `auth.js`
- 会话 `rt_session = { a: 账号, exp: 过期时间戳 }`，存 `localStorage`（记住）或 `sessionStorage`（临时）。
- `getSessionAccount()` 校验 `exp`，过期清除并返回 null。
- `navTo()` 压当前页入返回栈（`rt_back_stack`）再跳转；`goBack()` 弹栈回上一页，兜底 `history.go(-1)` → 首页。
- `getUserAsync()` 异步取当前用户信息；`getCurrentUserAccount()` 供写库函数取操作人。

### 3.4 离线层 `sw.js` + `manifest.json`
- `manifest.json` 提供 PWA 名称/图标/启动项；`sw.js` 缓存静态资源。
- 缓存策略：HTML 以 `fetch(req, {cache:'no-store'})` 始终取最新；其余资源 stale-while-revalidate。新 SW 安装后首次切换才生效，必要时硬刷新。

## 4. 业务模块设计（已上线）

### 4.1 登录与会话
- `login/classic.html`：账号/工号 + 密码（SHA-256）；"记住登录"与登录时长（24h/7d/30d）决定存储位置与 `exp`。
- `users.js` 的 `ensureDefaultAdmin()` 幂等播种 `admin/123`；`migrateAccounts()` 把旧 `localStorage` 账号导入 `users` 表（一次）。

### 4.2 首页看板 `app.js`
- **双区结构**：明确渲染「需求任务」区与「待办」区；待办区内按 `TODO_TYPE` 切换三类型。
- **卡片渲染**：`buildTodoCard` / `buildRequirementCard` 按类型布局不同 meta 字段；状态流转时间线灰显单行（`getStatusOpLine` + `TODO_OPERATION` 字典映射）。
- **操作按钮委托**：卡片按钮 `data-todo-act`，事件委托调用 `TODO_ACTION_HANDLERS[act]`；状态相关按钮由 `getTodoActions(statusCode, typeCode)` 动态生成（三类型按钮集不同）。
- **筛选**：类型/状态/优先级/项目/关键字；`renderTodoList`/`renderRequirementList` 各自拉取生命周期流水后批量渲染。

### 4.3 需求任务模块（`requirement-tasks.js` + `task-lifecycles.js`）
- **CRUD**：`createRequirementTask`/`updateRequirementTask` 先异步校验字典 code（任务类型/优先级/任务状态）与外键（项目/版本/开发人员），再开写事务；`deleteRequirementTask` **级联清理**图片/附件（媒体库）与 `task-lifecycles` 流水。
- **字段**：任务名称/描述/类型/优先级/状态/项目/项目版本/多开发人员/禅道 ID/图片附件/生命周期时间人/审计（详见 `DB_SCHEMA.md` 与需求说明书 §3.3）。
- **生命周期状态机**：`TASK_OPERATION`（DEV_SUBMIT/TEST_START/PAUSE/RESUME/TEST_DONE/ONLINE/RESET/DELETE），每次流转写 `task-lifecycles`（append-only）。

### 4.4 待办模块（`todos.js` + `todo-lifecycles.js`）
- **三类型状态机**：`TODO_TYPE` = 任务事项（TD_*）/ 缺陷（BUG_*）/ 会议（MT_*），各类型状态码与按钮集不同（由 `getTodoActions` 定义）：
  - 任务事项：开始处理/完成/编辑/删除/重置
  - 缺陷：开始处理/完成/**转交**/**上线**/编辑/删除/重置
  - 会议：开始/**取消**/**结束**/编辑/删除/重置
- **流转流水** `todoLifecycles`：每条 `todoId/statusCode/operationCode/operator/operateTime`（append-only），用于时间线与状态标签；`getStatusOpLine` 按当前状态取最近非编辑流水。

### 4.5 基础数据模块（共通模式）
- 每模块遵循统一模式：`registerStore` → `validateX`（同步格式校验）→ `createX/updateX/deleteX`（先跨连接异步校验外键/防环，再开写事务 + `try/finally` 统一 `db.close` 防连接泄漏）→ `getX/getAllX` + 层级构建（`buildHierarchy` 邻接表，`isDescendant` 防环）。
- `basic-data.html` 以 `MODULES` 数组罗列入口；待开发追加角色管理/权限管理。

### 4.6 统计报表
- `report.html` 汇总 + 四子页（任务/缺陷/待办/会议）；`report-common.js` 提供共享渲染 `buildTodoCardHtml` 与字典预取（`TODO_OPERATION_CODE_TO_NAME`、状态映射），各子页按筛选统计。

### 4.7 个人中心 / 存储备份 / 关于
- 个人信息 `profile*.html`（`updateProfile` 按需更新单字段）；账号安全 `security.html`；我的状态 `status.html`（抽屉展示）。
- `storage-backup.html`：导出/导入 IndexedDB 备份文件。
- `about.html`：关于 + `CHANGELOG.md` 更新日志 + 文档入口。

## 5. 权限系统设计（待开发）

### 5.1 设计目标与原则
- 实现 RBAC：角色聚合权限，人员赋予角色，按钮级 + 页面级 + 数据级（部门）三维控制。
- 单一真相：`PERMISSION_REGISTRY` 为权限 code 唯一来源；`menus` 表为其镜像 + 用户扩展。
- 历史可审计 + 运行时高性能（双轨）。

### 5.2 四表数据模型（SCD-2 追加写 + 去范式化双轨）
- `roles`：id, roleName, `menuCodes[]`（去范式化当前权限）, isSystemAdmin, enabled, 审计。
- `menus`：id, `menuCode`（业务编码/判定键）, menuName, `parentCode`（→父 `menuCode`，D1 确认）, nodeType(module/page/op), enabled, 审计。
- `role_permission`（历史）：id, roleId, `menuCode`(+menuId 镜像), snapshotId, 审计 —— **仅追加写**，保存时生成新 snapshotId 批量插入，旧行不 UPDATE/DELETE。
- `user_role`（历史）：id, userId, roleId, snapshotId, 审计 —— 同追加写。
- **双轨**：历史表只作审计日志；当前态在 `roles.menuCodes` / `users.roleIds` 覆盖，运行时 O(1) 解析，无 JOIN。
- `users` 表新增 `roleIds[]`（当前生效角色集）。

### 5.3 权限注册表 `PERMISSION_REGISTRY`（`permissions-registry.js`）
- 树形 `module > page > op`；命名全 **snake_case**（下划线，D 决策），如 `mod_basic`/`page_company`/`op_company_delete`/`op_task_dev_submit`/`op_todo_complete`。
- **看板拆「任务 / 代办(按类型)」**：`page_board_task` + 三个 `page_board_todo_*`（任务事项/缺陷/会议），各类型 `op_todo_*` 按钮集不同（对应 `getTodoActions`）。
- **报表拆四子页**：`page_report_task/bug/todo/meeting` 各带 `op_view/op_export`。
- **叶子 = 单个按钮**：集合中的按钮各自独立叶子，可单独勾选（粒度到按钮）。
- `isCodeConfigured(code)`：`menuCode ∈ 注册表展开集合` ⇒ 已配置（代码真正守卫）；否则"新增未真正配置"。
- `seedMenusFromRegistry()`：按注册表幂等播种 `menus`（`parentCode` 链式、`menuCode` 去重），用户可增删/启停。

### 5.4 运行时解析 `RT_PERM`（`permissions.js`）
- API：`can(account, code)` / `canAny` / `canAll` / `getMenuCodes(account)` / `isAdmin(account)` / `getDataScope(account)` / `guard(root)`。
- `can` 逻辑：`isAdmin` 短路 true → 否则取用户 `menuCodes`（users.roleIds→roles.menuCodes 去重展开）且对应 `menu.enabled===true` → 命中返回 true（**停用优先**）。
- 会话缓存：登录成功与角色变更时 `cachePermissions(account)`，避免每次判权读库。

### 5.5 `data-perm` 守卫（最小侵入接线）
- 各页按钮/入口加 `data-perm="op_xxx"`；渲染后调用 `RT_PERM.guard(root)` 统一隐藏无权限元素；条件逻辑用 `RT_PERM.can()` 显式判断（如空态"无权限"）。

### 5.6 数据权限（部门子树）
- `getDataScope(account)`：以用户 `departmentId` 为根，递归 `departments.parentId` 取部门子树（自身+下级，`includeSub:true`，D2）。
- 过滤落点：各数据层 `getAll*` 读取函数内按 `deptId ∈ 子树` 过滤（非仅 UI 隐藏）。
- **管理员豁免**（D6）：admin/系统管理员角色跳过过滤，可见全部。

### 5.7 系统管理员默认角色播种
- `ensureDefaultAdminRole()`（幂等）：建"系统管理员"角色（isSystemAdmin, menuCodes=注册表全 code），绑定 admin.roleIds；与 `ensureDefaultAdmin`/`migrateAccounts` 同位置触发。

### 5.8 停用优先语义
- `menu.enabled=false` ⇒ 该权限全局不生效（高于角色分配），用于紧急情况下线某操作（如禁用所有删除按钮）。

### 5.9 强规则（长期）
- 新增页面/按钮必须登记 `PERMISSION_REGISTRY` 并接线 `data-perm`；`release.sh` 发版自检扫描 `data-perm` 取值，未登记注册表则阻断发版；规则写入 `RULES.md`。

### 5.10 已确认决策（D1–D6）
- D1 `父级编号` 用 `menuCode`（主动偏离 RULES 外键约定，仅限菜单树）；D2 含下级部门；D3 仅 admin 不考虑老用户；D4 角色仅无人员引用可删 + 展示引用的人；D5 停用优先；D6 管理员最高权限看全部。

## 6. 数据设计概览
- **主库 `req-tracker`**（12 表）：companies / positions / departments / users / projects / projectVersions / dict / requirementTasks / taskLifecycles / todoLifecycles / todos / changelog。
- **媒体库 `req-tracker-pwa`**：images / attachments（Base64，按短 id 关联）。
- **权限新增 4 表**：roles / menus / role_permission / user_role（§5.2）。
- **外键策略**：统一存 32 位 ID；仅 `menus.parentCode` 指向 `menuCode`（D1 例外）。
- **索引策略**：按查询路径建索引；外键字段建索引；`developerIds` 用 `multiEntry`；历史表按 `roleId/userId/snapshotId` 建索引。
- 完整字段/索引/约束见 `DB_SCHEMA.md`。

## 7. 接口与交互设计
### 7.1 页面导航与返回栈
- 下钻入口统一 `navTo(url)`（压栈）；返回 `goBack()`（弹栈）；侧边栏/返回按钮遵循此机制。

### 7.2 登录闸门
- 各页 `<script>` 段 `if (!getSessionAccount()) location.replace('login/classic.html')`；首页/个人信息校验 `exp` 过期跳登录。

### 7.3 权限接线约定
- 受控元素加 `data-perm="<snake_case code>"`；`RT_PERM.guard()` 在渲染后统一隐藏；`RT_PERM.can()` 用于条件展示。

## 8. 非功能性设计
### 8.1 离线 / 缓存
- PWA 安装 + `sw.js` 缓存；HTML `no-store` 取最新，静态资源 stale-while-revalidate；`?v=` 破坏旧缓存。

### 8.2 性能
- 权限解析去范式化（`menuCodes`/`roleIds`）+ 会话缓存，避免每次渲染扫历史表 JOIN。

### 8.3 安全边界
- 密码 SHA-256 不存明文；会话带过期。**权限/数据权限为前端软隔离**：无后端，数据权限是前端只读过滤，不能替代服务端鉴权（对直接读 IndexedDB 的用户无效），计划中已明示。

### 8.4 兼容性
- 移动端优先：去除 `-webkit-tap-highlight-color`、保留 `:focus-visible`；返回用返回栈兼容华为浏览器。

## 9. 关键设计决策与权衡
| 决策 | 选择 | 理由 |
|---|---|---|
| 主键 | 32 位 `genId()` 十六进制 | RULES 强制，定长、无碰撞语义 |
| 历史表 | 追加写 + 去范式化当前态（双轨） | 既保留完整历史（审计），又运行时 O(1) 解析（性能） |
| 菜单树关联 | `parentCode` → `menuCode`（D1） | menuCode 是稳定业务标识，比随机 id 更适合树关联；主动偏离 RULES，仅限菜单树 |
| 已配置判定 | 以 `PERMISSION_REGISTRY` 为唯一真相 | 区分"代码真正守卫"与"用户新增未接线" |
| 停用优先 | 权限节点停用 > 角色分配 | 全局开关，紧急下线操作无需逐角色改 |
| 数据权限 | 部门子树 + 管理员豁免 | 满足"只看本部门数据"，管理员不受限 |
| 软隔离 | 前端只读过滤 | 无后端条件下的现实边界，不伪装硬安全 |
| 强规则 | 新增页面/按钮必登记权限 + 发版自检阻断 | 防权限漏登记，长期保持完整性 |
