# 会议时间流程化（修复「一律会议开始时间」bug）—— 批次 78 起

## 背景与根因

批次 73–77 上线后，用户核验发现**会议卡片时间不对**：

- 当前 `getStatusOpLine` 的会议分支把 `meetingTime` 字段**硬编码**为时间值，且 `opCode` 一律塞成 `TODO_START` 哨兵；
- 渲染层据此**无论会议处于什么状态，都标成「会议开始时间」**。

导致：
- 会议中 / 已结束 / 已取消 三态都显示**同一个 meetingTime 值 + 同一个「会议开始时间」标签**；
- 没有「会议结束时间」、没有「会议取消时间」；
- 显示的也不是「对应流程时间」（用户原话：*不是对应流程时间，只是会议开始时间*）。

首页待办卡片（app.js）与统计报表清单卡片（report-common.js）**都受影响**（两者都消费 `getStatusOpLine`），故「待办和统计都不对」。

## 确认：会议流转确实写了流水记录（数据可行）

`app.js` 的 `TODO_ACTION_HANDLERS` 与各创建路径：
- 创建会议 → `createTodoLifecycle({ operationCode: 'TODO_CREATE', statusCode: <初始> })`（行 1818-1821）
- 开始会议 → `TODO_START` / `MT_IN_PROGRESS`（行 2675）
- 结束会议 → `TODO_END` / `MT_ENDED`（行 2731）
- 取消会议 → `TODO_CANCEL` / `MT_CANCELLED`（行 2753）

即会议全流程（创建/开始/结束/取消）均落流水，完全可以走与普通类型一致的「取当前状态对应操作流水」逻辑。

## 正确口径（会议时间流程化）

`getStatusOpLine` 对会议**不再硬编码 meetingTime**，改为与普通类型一致：
- 初始态（MT_NOT_STARTED）→ `{ createdAt || CREATE流水.operateTime, 'TODO_CREATE' }`
- 非初始 → 取 `statusCode===当前 && operationCode!=='TODO_EDIT'` 的最近流水记录 → `{ operateTime, opCode }`
- 兜底 → `{ createdAt, 'TODO_CREATE' }` 或 `null`

渲染标签（**会议专属文案**）按 `opCode` 映射：
| opCode | 会议标签 |
|---|---|
| TODO_CREATE | 创建时间 |
| TODO_START | **会议开始时间** |
| TODO_END | **会议结束时间** |
| TODO_CANCEL | **会议取消时间** |
| 其它（兜底） | `opNameMap[opCode] + '时间'` |

> 注意：`meetingTime` 字段不再用于灰色单行（改走流程时间）；其彩色 meta「时间：meetingTime」保持不变。

  ```js
  // 会议专属标签：会议开始/结束/取消时间；其余回落通用
  var MEETING_OP_LABEL = { TODO_CREATE: '创建时间', TODO_START: '会议开始时间', TODO_END: '会议结束时间', TODO_CANCEL: '会议取消时间' };
  // opLabel 计算：
  if (t.typeCode === 'MEETING' && line) {
    opLabel = MEETING_OP_LABEL[line.opCode] || ((opNameMap[line.opCode] || line.opCode) + '时间');
  } else {
    opLabel = (line && line.opCode ? (opNameMap[line.opCode] || line.opCode) : '创建') + '时间';
  }
  ```
